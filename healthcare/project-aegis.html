Problem Decomposition — Where Do the Six Months Actually Go?

PHASE · FOUNDATION · COMPLETED
The Starting Question

Before designing anything, I needed to understand the actual time distribution inside the 6–8 month window. This is not an academic exercise — get the bottleneck diagnosis wrong and you build an AI system that optimizes the wrong thing. The most dangerous scoping error in Life Sciences AI: confusing protocol manufacturing time with trial execution time.

Project Aegis is about compressing the document manufacturing pipeline — drafting, reviewing, complying, localizing — not the clinical trial itself. Patients still need to be enrolled, treated, and observed. That is biologically constrained and no AI system changes it. Any architect who conflates the two has lost the room immediately.

Common Misconception to Discard
DISCARD — "Study conduct and monitoring takes too long."
Study conduct is the clinical trial itself — the period when patients receive the investigational drug and are monitored. This phase spans years, not months, and is governed by biology, not bureaucracy. AI cannot compress patient follow-up periods. The 6–8 month figure the Client cited refers exclusively to the time required to produce a submission-ready protocol document before a single patient is enrolled. Mixing these two timelines in front of a steering committee is a credibility-ending error.

The Insight That Changes the Architecture

Phases 2 through 5 — drafting, internal review, SAP alignment, and submission prep — run serially today. Each phase waits for the previous to complete before starting. The cumulative wait time from handoffs alone accounts for roughly 30–40% of the total calendar duration, independent of actual work effort.

This means the AI system's primary compression lever is not writing faster — it is parallelizing review domains simultaneously while a draft is being generated. A system that produces a draft in two days and then routes it through six-week sequential reviews has not solved the problem. The orchestration pattern matters more than the generation speed.

Architect's Position: "The 6–8 months is not a content problem — it is a coordination and serialization problem. An autonomous multi-agent system compresses time by running review domains in parallel against a live draft, not merely by writing faster."

Agent Decomposition — Who Does What?

PHASE · COMPLETED
Pattern Decision

The orchestration pattern is hierarchical supervisor/worker with parallel review bands. A single Orchestrator decomposes the protocol manufacturing pipeline into phases, assigns work to domain agents, and enforces hard blocking gates at regulatory milestones. Within the review tier, all critic agents run simultaneously against the same draft — not sequentially. This is where most of the calendar compression happens.

The pattern is not a pure pipeline because drafting and review overlap: sections can enter review before the full document is assembled. It is not a swarm because unconstrained peer-to-peer agent communication in a GxP environment creates an unauditable mess. Supervisor/worker with explicit gate nodes gives you the speed of parallelism with the governance of a controlled process.

The Decision That Changes Everything
The Protocol Ledger Agent is not optional. With 15+ agents reading and writing concurrently to a single document, you need one agent that owns the canonical version at all times. Without it, a Biostatistics Critic edit and a Regulatory Affairs Critic edit made simultaneously to the same section produce a conflict with no resolution authority. The Ledger Agent is the source of truth — every agent writes through it, never around it.

Design Principles Behind the Topology

One agent, one responsibility. The Protocol Design Architect role that initially seemed necessary dissolves into the Orchestrator when you're precise about what each owns. Command ambiguity between two high-level agents is a coordination failure waiting to happen.

Configurable over proliferated. Four separate section drafting agents (Clinical Objectives, Population, Treatment, Safety) sounds thorough, but sections are not independent — eligibility criteria constrain safety monitoring requirements which constrain endpoints. A single Section Drafter with domain-specific configuration and shared document context keeps consistency without the synchronization overhead of four agents writing to the same document.

Gate nodes are architecture, not process. The Human Checkpoint Gate is a first-class agent node, not a footnote. In a GxP environment, if human approval is not modeled as a blocking dependency in the workflow graph, it gets bypassed under schedule pressure. Making it explicit in the topology makes it auditable and enforceable.

Steering committee framing: "We are not building a chatbot that writes protocols. We are building a document manufacturing system with parallel quality control bands, an authoritative state layer, and hard human gates at regulatory milestones. The orchestrator sequences; the critics run simultaneously; the ledger holds truth; humans approve at defined checkpoints. Nothing clears a gate without a named human sign-off."

Why Domain-Specific Language Models Matter for This Architecture

SIDEBAR · MODEL SELECTION · GARTNER INSIGHT
The Model Layer Decision

Selecting the right model tier for each agent class is not a procurement footnote — it directly determines accuracy, cost, and regulatory defensibility at scale. Gartner's emerging research on Domain-Specific Language Models (DSLMs) is directly applicable to the agent topology designed in Step 2.

DSLMs are generative AI models trained and fine-tuned on specialized corpora for a specific domain — regulatory, biomedical, legal, cybersecurity. Unlike general-purpose frontier models trained on broad web data, DSLMs internalize domain vocabulary, regulatory nuances, and workflow context at a structural level. Crucially, they tend to be smaller (commonly 7B–13B parameters), cheaper to operate, and measurably more accurate on domain-specific tasks than large general models that guess at specialized terminology.

For Project Aegis, with 18 agents running across potentially thousands of protocol generation cycles, the cost and accuracy argument is not academic. A 7B regulatory DSLM running inference on a Critic Agent cohort costs a fraction of GPT-4-class API calls and produces fewer domain errors — which in a GxP context translates directly to fewer compliance failures.

DSLM Applicability Across the 18-Agent Topology
DSLM Advantage Why It Matters Agents That Benefit Directly
Higher accuracy in domain language Regulatory critics must produce precise ICH/FDA/EMA citations and MedDRA-coded adverse event language. General models approximate; DSLMs are trained on the source material. Regulatory Affairs Critic, Medical Critic, Safety/PV Critic, Translation & Terminology Agent
Lower hallucination rate on domain facts A fabricated stopping rule or misquoted guideline section is a patient safety event. DSLMs hallucinate less on in-domain content because the training distribution matches the task. Compliance Guardrail Agent, Protocol Ledger Agent, Section Drafter Agent
Deterministic, explainable outputs GxP validation (21 CFR Part 11) requires that outputs be traceable to a documented knowledge source. DSLMs produce more consistent, less stochastic outputs on in-domain tasks. Audit & Traceability Agent, Human Checkpoint Gate, Regulatory Packaging Agent

Recommended Model Assignment

Biomedical/Regulatory DSLM — base model for all drafting, evidence, and statistics agents. Trained on PubMed, ClinicalTrials.gov, FDA guidances, and ICH documents. Handles generation tasks where domain fluency and evidence synthesis are primary requirements.

Regulatory Compliance DSLM — assigned to all Critic agents and the Compliance Guardrail. Fine-tuned specifically on regulatory decision corpora: agency rejection letters, approval reviews, EMA assessment reports, and compliance audit findings. This model needs to reason about what is wrong in a document, not just what is right.

DSLMs are not a replacement for retrieval. The knowledge architecture from Step 3 — segmented RAG indices, the regulatory knowledge graph, and the citation ledger — remains the authoritative runtime knowledge layer. The DSLM provides the reasoning backbone; retrieval provides the current, citable facts.

Knowledge Architecture — What Does the System Need to Know?

PHASE · COMPLETED
The Core Question

Eighteen agents operating in a GxP-regulated environment need knowledge that is current, attributable, and structurally accurate. RAG gets you most of the way — but three things a vector store alone cannot give you are what separate a defensible protocol from a hallucinated one: a knowledge graph for regulatory dependency traversal, a citation ledger for version-pinned provenance, and a grounding enforcer that treats unattributed claims as compliance violations.

Parametric vs. Retrieval Knowledge
TypeDefinitionExamples in AegisTrustworthy for Compliance?
ParametricBaked into model weights; static after trainingICH E6(R3) structural logic, GCP principles, statistical reasoning patterns, medical writing conventionsNo — frozen, unattributable
RetrievalFetched at runtime from versioned external sourcesFDA guidance published last month, the Client SOP v4.2, Compound X 2024 trial data, Brazil 2025 regulatory deltaYes — current, citable, auditable

Retrieval knowledge dominates for compliance purposes. Every factual claim in the protocol must cite a specific source, section, and version — parametric knowledge cannot satisfy this requirement. The model's backbone provides reasoning capability and structural understanding; runtime retrieval provides the authoritative, datestamped facts that go into the document.

Use a domain-fine-tuned model (biomedical + regulatory LLM) as the parametric backbone. Never rely on it for current regulatory rules, internal SOPs, or compound-specific data. Always retrieve those at runtime.

Three Things RAG Alone Cannot Do

1. Regulatory dependency traversal. ICH E6(R3) §5.1.1 depends on §4.3 which is referenced by FDA guidance §2.1. A flat vector store chunks these into isolated semantic units and loses the cross-reference. When the Regulatory Affairs Critic asks whether a safety monitoring section is compliant, it needs to know that §8 (essential documents) structurally constrains §5 (sponsor responsibilities). A Regulatory Knowledge Graph (Neo4j or equivalent) layered over the vector indices handles this — nodes are sections and guidances, edges are cross-references, supersession chains, and temporal versioning. Vector search handles "what's relevant"; the graph handles "what depends on what."

2. Regulatory versioning as change-impact, not re-indexing. When FDA issues a new guidance, the instinct is to re-index. The harder problem: which already-generated protocol sections were cited against the old version? A Citation Ledger — an append-only store where every agent claim carries a pointer to the specific document version it was grounded in — enables the system to flag all sections citing version N when version N+1 is ingested. This is the Amendment Impact Agent's function extended into the knowledge layer.

3. Grounding enforcement. Agent outputs that cannot be attributed to a retrieved source chunk must be blocked, not just flagged. The Compliance Guardrail Agent maintains a citation manifest against which every agent assertion is validated. Unattributed claims in a GxP environment are not a quality issue — they are a compliance violation.

Why inline-only is insufficient in GxP: LLMs can drift during multi-turn generation even when the initial prompt was valid. A section that passes inline grounding at turn 1 may introduce a non-compliant cross-reference by turn 4 as context accumulates. Post-generation validation is not optional — it is the safety net for everything the inline layer cannot see.
What the Human Actually Validates

The human checkpoint in GxP is a signatory obligation under 21 CFR Part 11, not a quality preference. The human is the accountable entity in any FDA audit. This means the checkpoint has two distinct and non-interchangeable gates:

PHASE · COMPLETED

Why conflating them fails: "Include informed consent in local language" is a regulatory requirement, not a translation task. If you treat it as translation, the protocol misses the requirement itself — not just the language. Amendment propagation also breaks: a global protocol change triggers regulatory re-localization for specific sections; it should not trigger full re-translation of unaffected content. Conflation produces unnecessary re-work and missed regulatory obligations simultaneously.
Localization Is Three Problems, Not One

The architecture must handle three distinct localization types, each with its own agent, knowledge source, and validation path:

Regulatory localization — changes what is in the protocol per jurisdiction. Governed by the delta database and handled by the Country Localization Agent. Requires regulatory expert review and human approval gate.

Linguistic localization — changes how content is expressed. Governed by Translation Memory, locale-specific MedDRA LLT mappings, and approved terminology glossaries. Requires back-translation validation on safety-critical sections (dosing, stopping rules, consent language). A single forward-translation pass is not sufficient for patient-facing or safety-critical text.

Format localization — changes the document package structure itself. FDA requires an IND (Investigational New Drug Application); EMA requires a CTA (Clinical Trial Authorization). These are structurally different submission packages — different cover pages, module structures, appendix requirements, and labelling formats. This is not a content delta; it is a template difference that the Regulatory Packaging Agent handles via jurisdiction-specific output schemas.

Delta Database Schema

The delta database is the knowledge source for the Country Localization Agent. It is a rule-based catalog — not flat text — structured as a graph to support cross-reference with the ICH section hierarchy.

Delta Record
├── jurisdiction          (FDA | EMA | PMDA | ANVISA | CDSCO | ...)
├── regulatory_domain     (Safety Monitoring | Consent | DSMB | Endpoints | ...)
├── delta_type            (ADD | REMOVE | MODIFY | REPLACE | FORMAT)
├── reference_section     (ICH E6(R3) §5.10.2)
├── delta_rule            (machine-readable constraint)
├── source_guidance       (citation + version + effective_date)
├── delta_version         (version of this delta record itself)
├── delta_status          (ACTIVE | SUPERSEDED | PROPOSED)
├── impact_scope          (which global sections are affected)
├── section_mapping       (global_section → local_section map per variant)
├── conflict_registry_ref (pointer to cross-jurisdiction incompatibilities)
├── approval_status       (approved by country regulatory team)
└── reviewer              (named human approver)
Conflict Registry: The delta database must pre-compute cross-jurisdiction incompatibilities. EMA may require a DSMB independence clause that PMDA prohibits for domestic trials. Without a Conflict Registry, the Country Localization Agent produces individually valid country variants that cannot be reconciled into a coherent multi-regional package — discovered only at submission stage.

When a global protocol amendment is issued, the architecture propagates changes to country variants without triggering full re-localization. Orchestration Pattern — Supervisor/Worker with Embedded Mini-Pipelines

PHASE · COMPLETED
Pattern Definitions — The Decisive Variable
PatternDefinitionWho Controls Task Decomposition?Verdict for Aegis
Pipeline Tasks flow through a fixed, linear sequence — each stage depends on the previous stage's output Predefined sequence — no dynamic routing REJECTED
Swarm Agents operate as peers, autonomously deciding when to call other agents based on local reasoning Each agent — fully decentralized REJECTED
Supervisor/Worker Central Orchestrator decomposes work into tasks, dispatches to specialized workers in parallel bands, and reconciles outputs Orchestrator — centralized with parallel execution SELECTED
Why Pipeline Fails

Pipeline enforces serial execution by design. Applied to Aegis, this means: Medical Critic completes → Biostatistics Critic starts → Regulatory Affairs Critic starts → Safety Critic starts → Legal Critic starts. Total review time equals the sum of all critic durations — potentially 10+ hours — instead of parallel execution at the duration of the slowest critic — approximately 2 hours.

Step 1 established that phases 2–5 running serially is the primary bottleneck producing the 6–8 month timeline. The single highest-value architectural decision in the entire system is the parallel critic layer. Pipeline structurally eliminates that parallelism. It is not a suboptimal choice — it is a direct inversion of the value proposition.

Why Swarm Fails

Swarm produces three specific failure modes that are incompatible with GxP operation: agents self-routing without central record (no auditable task decomposition); circular calls creating infinite loops with no coordinator to detect and break them; and concurrent writes to the Protocol Ledger without coordination, producing version corruption.

The deeper issue is validation. 21 CFR Part 11 requires a documented, reproducible execution path. A swarm's execution paths are emergent — they depend on each agent's local reasoning at runtime and cannot be predefined or validated as a stable process. FDA inspection readiness requires the ability to demonstrate that the system follows a documented workflow. Swarm cannot satisfy this requirement by design.

What "Embedded Mini-Pipelines" Actually Means

The pattern is hierarchical: the macro-level structure is a pipeline enforced by the Orchestrator (Research → Drafting → Assembly → Review → Submission run sequentially — each phase depends on the prior), while the micro-level structure within each phase is supervisor/worker with parallel workers.

Within the review phase specifically, the sequence is: five critics execute in parallel → Orchestrator detects conflicts → Resolution Agent proposes reconciliation → Human Gate blocks until approved. That three-step intra-phase sequence is a mini-pipeline running inside the supervisor/worker phase. This is the structural distinction from pure supervisor/worker, and it matters because the mini-pipeline enforces the ordering of conflict resolution steps that GxP requires.

State Management — Protocol Ledger Design

The core state management problem: 18 agents reading and writing to the same document simultaneously produces version conflicts. Section-level locking prevents most conflicts but introduces deadlock risk — Medical Critic holds §5.1 and requests §5.2; Biostat Critic holds §5.2 and requests §5.1. Classic circular deadlock.

Three design decisions resolve this: lock ordering (all agents acquire locks in ascending section number order, eliminating circular wait); field-level ownership tagging (statistical claims, regulatory citations, and safety requirements within the same section are owned by different agent classes and written to independently — reducing conflict surface by an order of magnitude); and durable execution for the Orchestrator (task graph persisted externally, not held in memory, so an Orchestrator crash at hour 72 of a 100-hour run restarts from current state rather than forcing a full replay).

Technology stack: PostgreSQL + pgvector for versioned document storage with append-only commits. Neo4j for cross-version dependency graph. Redis for section-level lock management with timeout-based auto-release. Temporal.io or AWS Step Functions for durable Orchestrator execution.
